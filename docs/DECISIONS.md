## ADR-024: Redesign v2 — three new signal types added after real SDR feedback that funding+hiring is "the basic approach"
**Decision:** Paused the n8n workflow entirely (excluded from active use pending a broader system-workflow redesign) and added three new signal-capture features on top of the existing pipeline: buying-intent language mining (`python/hiring_ats_lookup.py`), leadership-page diffing (`python/leadership_monitor.py`, new), and Product Hunt launch monitoring (`python/producthunt_launches.py`, new — Branch D).
**Why:** An SDR reviewing the build pointed out that funding events and hiring volume are exactly what ZoomInfo/Apollo/every off-the-shelf sales-intelligence tool already sells — building a free version of that isn't differentiated "in-house tooling," it's re-deriving the commodity layer. Rather than guess at what would be more creative, ran the "Trigger-Bot" prompt (`notes/GTM signal engine prompt....md`) against freshly-researched real Productboard data (their actual named customers, integrations, competitor pricing — not assumption) to get a structured, ranked list of buying-trigger categories, then cross-referenced it against what this project already captures vs. what's genuinely missing. Full research trail: `redesign/01-trigger-prompt-filled-productboard.md`, `redesign/02-signal-intent-catalog.md` (32 signal types across 7 categories, tagged by buildability), `redesign/03-creative-signal-approaches.md` (9 ranked ideas).
**Real finding from that research, worth naming:** Productboard's own named customers (Autodesk, Salesforce, Zoom, Medtronic) are enterprise-scale, not just 50-500-employee startups — broader than the original blueprint's ICP assumption. Not acted on in this round (out of scope), but flagged as a real open question for whoever revisits scoring.py's demographic bucket.
**Why these 3, in this order:** ranked by payoff-to-risk ratio, not novelty alone. Buying-intent mining reuses 100% existing infrastructure (Branch B's already-fetched job description text, `classify.py`'s already-proven Gemini pattern) — zero new external dependency, zero new risk. Leadership-page diffing closes a real gap (new decision-maker detection) without inheriting the LinkedIn-scraping risk this project has twice already deliberately avoided (ADR-009/010) — the source is a company's own public marketing page, not a third party with ToS restrictions. Product Hunt fills the other flagged gap ("new product launch") via a confirmed-free, official API, mirroring the trust profile that made EDGAR/Adzuna/Greenhouse/Lever load-bearing branches instead of narrative-only ones. Two lower-ranked ideas from the catalog (employee-review mining, domain-monitoring) were deliberately NOT built this round — they carry real, undiscounted risk (a third scraping-posture decision after LinkedIn/G2) or too low a hit rate to prioritize yet.
**Full planning discipline before any code was written:** ran in Claude Code plan mode — 3 parallel Explore agents read the exact existing patterns to extend (Gemini classification shape, Branch B job-description flow, scoring/schema/API conventions), 1 Plan agent produced a detailed design flagging 8 open questions, then explicit user sign-off was obtained on the 3 real architectural decisions (buying-intent call frequency, leadership monitoring's direct-DB-write exception, beautifulsoup4 vs. regex) before implementation began. Plan preserved at `~/.claude/plans/let-s-continue-with-your-reflective-sifakis.md`.
**Explicit exception documented, not glossed over:** the new `/leadership/run` endpoint writes directly to Postgres, unlike every other branch endpoint (which only land raw JSON, leaving all DB writes to `/merge/run`). Justified because leadership-page diffing needs durable snapshot state (`company_leadership_snapshots`, new table) to compare against on the next run — a one-shot land-then-merge flow doesn't support "did this change since last time." Named as a deliberate architectural exception in both the endpoint's docstring and this ADR, not a quiet inconsistency.
**Real, honestly-flagged uncertainty carried forward, not resolved yet:** Product Hunt's product names are brand names, not necessarily registered legal company names — structurally more likely to miss/duplicate-match against the `companies` table than Branch A/B (both of which use formal names). No assumption made about the real match rate; same treatment as Branch B's measured ~7% ATS-slug-guess hit rate (docs/ISSUES.md) — needs a live test to actually know.
**Status:** Decided and built, not yet live-tested. All three features are mocked-tests-only so far (35 new pytest cases, 202/202 passing) — no real Gemini calls, no real Product Hunt calls, no real leadership-page fetches have happened yet. Live verification is the explicit next step before any of this feeds real scoring decisions.

## ADR-022: api/main.py extended into the full n8n-facing API surface; n8n orchestrates, doesn't reimplement
**Decision:** Extended `api/main.py` (previously just `/score`) with `POST /branch-a/run`, `/branch-b/run`, `/branch-c/run`, `/merge/run`, and `/pipeline/run-all` - thin wrappers around already-tested Python (funding_edgar, hiring_signals, intent_g2, merge_signals, pipeline). n8n's workflow (built by the user in the UI, ADR-007) calls these in sequence; it does not reimplement any branch logic, merge logic, scoring, or outreach generation itself via raw HTTP/Code nodes.

**Real bug caught by live-testing the actual deployed shape, not just pytest's TestClient:** `api/main.py` never called `load_dotenv()` - a fresh `uvicorn` process had no environment variables and every endpoint touching a credential failed with a `500`. Fixed; full writeup in `docs/ISSUES.md`.
**Live-verified 2026-07-12** with a real standalone `uvicorn` process (not reusing any warm Python session): `/branch-a/run` (22 real companies), `/branch-b/run` (40 real companies, a fresh live run), `/merge/run` (62 distinct companies), `/pipeline/run-all` (62 evaluated, all correctly `not_qualified` - consistent with the earlier full-database scoring finding). `/branch-c/run` confirmed only via its mocked unit test, not live-called, given real per-call cost.
**Status:** Decided and live-verified (except Branch C's live call, deliberately deferred to the user per ADR-009). 3 new `utils/db.py` functions + tests, 8 new/updated `test_api.py` cases, 4 new `test_pipeline.py`/`test_db.py` cases for the dedup-window check built alongside this.

## ADR-021: Branch C rearchitected to accumulate reviews over time, not replace them each run
**Decision:** `competitor_intel` is no longer written directly from each scrape's local aggregate. A new `g2_reviews` table stores every individual review (keyed `UNIQUE` on `review_id`), and `competitor_intel` is now a value RECOMPUTED from that table's full history after each merge - `merge_intent_signals()` upserts each new review (deduped, so re-fetching an already-seen review is a harmless no-op) then recomputes the touched competitor's aggregate from scratch via `db.get_competitor_intel_aggregate()`.
**Why:** the user clarified Branch C's intended real-world usage (2026-07-11): it should run periodically (not necessarily daily, given real Apify cost) and "capture new/recent reviews if given," updating the dataset incrementally rather than wholesale re-scraping. The prior design (`merge_intent_signals()` directly writing whatever the current batch's `pain_point_corpus` computed) would have LOST all prior history on every subsequent run - a periodic run pulling only recent reviews would silently overwrite `total_reviews_seen`/`switch_signal_count`/etc. with just that batch's numbers, discarding everything accumulated before. Caught and fixed before this became a real scheduling bug, not after.
**Why DB-level dedup instead of an actor-level "since" cursor:** the Apify actor's documented parameters don't support fetching only reviews since a given date/id. Rather than build that missing capability, `review_id UNIQUE` + `ON CONFLICT DO NOTHING` at the database layer makes ANY overlapping re-scrape safe by construction - simpler, and doesn't depend on actor behavior we don't control.
**Contract change:** `intent_g2.get_intent_signals()`'s output shape is unchanged (`{"reviews": [...], "pain_point_corpus": {...}}`), but `merge_signals.merge_intent_signals()` and `run_full_merge()` now consume the `reviews` list (individual, `review_id`-tagged), not the `pain_point_corpus` dict - the batch-local `pain_point_corpus` is still returned for a quick immediate-run summary, but the DB-recomputed aggregate is now the single source of truth for `competitor_intel`.
**Live-verified 2026-07-11** against the real 60-review dataset: merged all 60 for the first time (4 competitors, 15 each) - `switch_signal_count` totals summed to 13 across all competitors, matching the exact figure documented earlier this session after the truthy-string bug fix (ADR-012), a real independent cross-check. Re-ran the identical merge - confirmed idempotent (still 60 rows, no duplicates, no count drift). Added one synthetic new review for Aha! and confirmed the aggregate correctly incremented (15→16 total, 6→7 switch signals) rather than resetting - exactly the "capture new, keep the rest" behavior the user asked for.
**Status:** Decided and live-verified. 4 new pytest cases (2 `test_db.py`, 2 `test_merge_signals.py`), plus 2 existing `test_merge_signals.py` cases updated for the new function signature (`normalized_reviews` replacing `pain_point_corpus`).

## ADR-019: Phase 8 outreach generation adds a priority_summary field; JSON-fence parsing shared with classify.py
**Decision:** `python/outreach.py` implements the blueprint's signal-specific outreach generation (TIMING/INTENT/BOTH angles, email subject A/B, email body, LinkedIn message, call script), plus one new field beyond the blueprint's original scope: `priority_summary`, a 1-2 sentence "why contact now" rationale synthesizing the lead's facts, added at explicit user request (2026-07-11) to support the CRM-style Google Sheet planned for Phase 9 - a human triaging a sheet of leads needs a scannable justification, not just a raw `score_breakdown` JSON blob.
**Why this connects Phase 5/Branch C to something real:** the INTENT and BOTH angles pull real G2 pain-point quotes from `competitor_intel` (via the lead's `competitor_pain_quotes`, sourced from the same corpus `scoring.py`'s INTENT bucket already scores) directly into the generated copy - this is the actual payoff of building Branch C's aggregate corpus and ADR-013's job-description tool-mention scanning: specific, evidence-backed messaging instead of generic "you might be unhappy with your current tool" copy.
**Shared JSON-fence parsing:** `python/classify.py`'s `_parse_json_response()` (built in Phase 5 to strip markdown code fences Gemini adds despite instructions not to) was moved into `utils/gemini.py` as `parse_json_response()`, since `outreach.py` needed the identical logic - a second real call site turning a one-off helper into shared infrastructure, not premature abstraction. `classify.py` re-imports it under its original private name to avoid touching its own already-passing tests.
**Fail-loud design choice:** unlike `classify.py`'s best-effort extraction (a bad Gemini response there just means "no company found," a safe no-op), `generate_outreach()` raises `RuntimeError` on an unparseable or incomplete response - outreach copy is customer-facing content that could actually get sent, so silently degrading to partial/garbage output is worse than failing loudly and re-running.
**Live-verified 2026-07-11** against real data: TIMING angle tested against the real "InfraSight Software Corp" row from Phase 6 (real funding stage/date) - correctly referenced the actual funding round and hiring activity in generated copy. INTENT angle tested with synthetic-but-realistic pain-point quotes (no real company has `current_tool_mentioned` set yet, since that requires a live Branch B run against real job postings) - correctly wove the specific quotes into the email/LinkedIn/call-script copy rather than generic messaging.
**Status:** Decided and complete. 9 pytest cases in `tests/test_outreach.py`, all mocked.

**Decision:** `utils/hubspot.py` implements the blueprint's Phase 5 dedupe logic (search by domain, check last-contacted recency, decide create/skip/update), with the API endpoint, filter shape, and the "last contacted" property all confirmed live against the real HubSpot sandbox before writing any code - not assumed from the blueprint or from documentation.
**Why verify live instead of trusting docs or the blueprint:** every external API touched this session (SEC EDGAR's field names, the G2 actor's `productUrls`/`switchedFromOtherProduct` shape, Gemini's model lineup) turned out to differ from an initial doc-based or blueprint-based assumption in some real way. A doc fetch for HubSpot's search API this same session returned a mismatched endpoint (`crm/objects/2026-03/companies/search`) that didn't match what actually worked when tested live (`crm/v3/objects/companies/search`) - confirming the pattern held here too.

**Status:** Decided and complete. 10 pytest cases in `tests/test_hubspot.py`, all mocked against the live-confirmed real shapes.

## ADR-017: Clay enrichment built as a CSV export/import round-trip, not a live webhook
**Decision:** Phase 6's Clay integration uses a batch CSV round-trip instead of Clay's live inbound/outbound Webhook triggers: export companies missing a domain from Postgres → import into a free-tier Clay Blank table → run waterfall enrichment columns → export the enriched result → a Python script loads it back into Postgres via `utils/db.py`'s `upsert_company()`.
**Why:** While setting up the Clay table (2026-07-10), discovered Clay's Webhook trigger (both as a Signals-category source and as an automated outbound action) is gated behind a paid plan upgrade on the user's current account. Consistent with this build's established pattern of avoiding new recurring costs where a free alternative exists (AWS dropped entirely rather than attach a card - ADR-003; Crunchbase swapped for free SEC EDGAR - ADR-008), chose the CSV round-trip over upgrading Clay.
**Tradeoff, stated plainly:** this is batch, not real-time - a human has to trigger the export, import, run enrichment, and export again each cycle, rather than Clay automatically enriching new companies as they're written to Postgres. For a portfolio pipeline that isn't running as a live 24/7 production service, this doesn't cost real functionality, just automation polish - an honest thing to name directly if asked in the interview ("why not the live webhook") rather than something to gloss over.
**Status:** Decided and complete. `python/enrichment.py` (export + import), plus `utils/db.py`'s new `update_company_by_id()`, fully live-verified 2026-07-11 - all 22 real companies in Supabase now have real domains via the CSV round-trip (user ran the Clay side; Claude verified both the export and import live). 5 pytest cases in `test_enrichment.py`, 2 in `test_db.py`.
**Real finding from the actual output, worth naming:** 2 of 22 domains look like waterfall-enrichment mismatches (a generic company name resolved to an unrelated real domain; a Webflow placeholder domain where no confident match existed) - see `docs/ISSUES.md`. Imported as-is, not silently corrected - an honest limitation of this class of tool, not something a smarter import script can fully solve without adding a validation layer that was out of scope for this pass.
**Also note:** this round only backfilled `domain` - `employee_count` and `is_saas` (needed for `scoring.py`'s demographic bucket, ADR-014) were never part of the Clay table's enrichment columns in this pass and remain NULL. Worth adding as a second enrichment column later if the demographic bucket needs to go live.

## ADR-016: Phase 6 and Phase 7 swapped - Clay enrichment now comes before the HubSpot dedupe check
**Decision:** Reordered the remaining phase sequence: **Clay enrichment (domain backfill) now runs before the HubSpot dedupe check**, reversing the original blueprint's order.
**Why:** While scoping the HubSpot dedupe check (originally Phase 6), the user pointed out there's limited value in building it now - its primary, reliable matching signal is `companies.domain`, which stays `NULL` until Clay enrichment (originally Phase 7) runs. Building the dedupe check first would mean writing and testing only the weaker name-based fallback path today, then having to properly exercise the real domain-based path later anyway once Clay exists - the same "build once, correctly, with real data" reasoning already applied to `scoring.py`'s demographic bucket in ADR-014.
**The original blueprint's ordering made sense for its own architecture** (Crunchbase was assumed to supply a domain far earlier in the pipeline) but stopped making sense once we deliberately deferred domain resolution to Clay (ADR-004/ADR-008's `normalized_name` interim-key design) - this is a case where a downstream architecture decision should have prompted revisiting an upstream phase order sooner, caught now rather than after building something whose main value was still blocked.
**New order:** Phase 6 = Clay enrichment (`python/enrichment.py`, webhook integration, domain waterfall). Phase 7 = HubSpot dedupe check (`utils/hubspot.py`) - built after, so its domain-based search can be tested against real, non-null domains from the start instead of a placeholder fallback.
**Status:** Decided. No code written for either phase yet.

## ADR-015: Phase 5 Gemini classification scoped to the one real gap - G2 review company attribution
**Decision:** `python/classify.py` does exactly one job: given a G2 review's free text, ask Gemini (via the new `utils/gemini.py` dual-key/backoff client) whether the reviewer explicitly named their employer, and extract it if so. Successful attributions flow into the normal per-company `signals` table via `merge_signals.py`'s new `merge_attributed_reviews()`.
**Why this scope, not the original blueprint's broader Phase 3 (sentiment/pain-points/urgency classification):** most of that no longer applies - Reddit was dropped entirely (ADR-012), and G2 review text is already structured enough (`star_rating`, `switch_reason`, `is_negative`, `is_switch_signal`) without an LLM. The one real, previously-deferred gap is company attribution, explicitly flagged in `intent_g2.py`'s own docstring as "a Phase 5 Gemini classification job, not attempted here with brittle regex" - regex isn't reliable for company-name extraction given how much real phrasing varies (legal suffixes, abbreviations, no mention at all).
**Why an LLM instead of regex here specifically:** unlike ADR-013's competitor-tool scan (a closed set of 4 known product names, well-suited to regex), a company name is an open-ended set with no fixed vocabulary - exactly the kind of nuanced natural-language judgment call regex handles poorly and an LLM handles well.
**Design details:** `extract_reviewer_company()` skips the Gemini call entirely for empty/blank review text (a cost guard) rather than sending a doomed API call. `_parse_json_response()` strips markdown code fences before parsing, since Gemini frequently wraps JSON in ```` ```json ```` fences despite explicit instructions not to - a real, common LLM-output quirk, not a hypothetical edge case.
**Live-test status (2026-07-10):** succeeded, after a real detour - see `docs/ISSUES.md` for the full story. The first attempt returned a `429` that looked like quota exhaustion; the user correctly identified it as a deprecated-model symptom instead (the originally hardcoded `gemini-2.0-flash`). Querying the live model list directly (`GET /v1beta/models`) found the real current lineup, and even the "obvious" successor (`gemini-2.5-flash-lite`) was already `404 - no longer available to new users`. `utils/gemini.py` now defaults to `gemini-flash-lite-latest`, a rolling alias, specifically to avoid a pinned model name going stale again. Confirmed against real G2 review text (0/15 attributed - expected) and synthetic text naming a company (correctly extracted). `GEMINI_API_KEY_2` remains invalid - user's explicit call: it's only needed as a fallback for when the primary key hits its quota, which isn't happening yet, so fixing it is deferred rather than treated as an open gap.
**Status:** Decided and live-verified. 11 pytest cases across `utils/gemini.py` and `python/classify.py`, plus 1 in `merge_signals.py` for the new merge function.

## ADR-014: Phase 4 scoring formula adapted from the blueprint to match what our branches actually produce
**Decision:** `python/scoring.py` implements the original blueprint's weighted ICP scoring model (TIMING/INTENT/DEMOGRAPHIC buckets + deductions + BOTH-signal bonus, threshold >=70), with three deliberate adaptations agreed with the user (2026-07-10) before writing any code:
1. **INTENT scoring uses `current_tool_mentioned` (ADR-013) scaled by that competitor's severity in `competitor_intel` (ADR-012)**, not a per-company G2 star rating - structurally impossible since G2 reviews are anonymous. Banded (not linearly scaled) against the one real dataset available: Aha!'s real 13/60 = 21.7% switch-signal rate anchors the "high" band (>=20%). A company whose tool can't be identified (untracked competitor, or no tool mentioned at all) scores 0 INTENT points - not a penalty, since absence of evidence isn't evidence of absence.
2. **The "posted Product Ops role" bonus excludes any title that also says "manager"** (e.g. "Product Operations Manager") - only an unambiguous dedicated-ops title (e.g. "Product Operations Lead") earns it, on top of that same posting still counting toward the general PM-posting-count bucket. Prevents one posting from double-counting the same underlying fact under two different bonus names.
3. **Demographic fit (employee_count, is_saas) is fully implemented and pytest-covered, but not wired into any live orchestration yet.** Every real company row has these fields NULL until Phase 7's Clay enrichment exists. Considered scoring now with a recalibrated threshold (partial data, adjust later) vs. deferring; user chose "score once" - build and test the complete function now (so nothing blocks later), but don't call it against live data until its inputs actually exist. Avoids the data-integrity problem where the same `icp_score` value would mean two different things before/after enrichment.
**Also not implemented:** the blueprint's "+5 no existing Product Ops role found" sub-bonus - no current data source can confirm whether a company already employs someone in that role (would need something like a LinkedIn people-search, dropped per ADR-010's safety review). Omitted rather than guessed; demographic bucket max is 25, not the original 30.
**Also dropped:** Reddit/LinkedIn intent signals - never built (ADR-012), so not scored.
**Status:** Decided. 15 pytest cases in `tests/test_scoring.py`, all passing; not yet live-tested against real Supabase data (see `docs/HANDOFF.md`).

## ADR-012: Branch C built on the Apify G2 scraper, purpose reframed as competitive-intel content
**Decision:** Branch C (intent signal) uses the Apify actor `automation-lab/g2-scraper` to pull reviews of named competitors (Aha!, Jira Product Discovery, ProductPlan, Craft.io). Per ADR-009, Claude writes and mock-tests the code but does not run live Apify calls itself - the user tests live to manage their own Apify usage.
**Alternatives checked first (2026-07-09), in order:** G2's own official Developer API and MCP server - both confirmed vendor-scoped (ADR-011), can't query competitor products at all. Capterra - no viable self-serve official API. Reddit - genuinely free official API, but not empirically tested for real coverage of these 4 competitor names given time constraints. Hacker News - free/official, empirically tested live: too noisy for "Aha!"/"ProductPlan" (common English phrases, not the product) and too low-volume for "Craft.io"/"Jira Product Discovery" to be usable.
**Why this actor specifically:** no login bypass (public review pages only), no proxy required (unlike other G2 actors that explicitly recommend residential proxies to evade anti-bot measures), built-in polite delays, minimal PII (reviewer display name only). A conscious, documented scraping tradeoff made only after the official alternatives above were checked and ruled out - not a default choice.
**Structural limitation discovered, and how it reshaped the design:** this actor's reviewer fields are name, country, region, company-size segment, and industry code - **not the reviewer's company name**. G2 reviews are largely anonymous by design. This means most reviews can't be attached to a specific company the way Branch A/B signals can, which matters because the scoring model's "BOTH signal" bonus depends on one company hitting both TIMING and INTENT. The original blueprint actually anticipated this - its Phase 1 spec listed the field as "company_name (if mentioned in review)," i.e., best-effort, not structured.
**Resulting two-part design (reframed per user's own articulation of G2's value - competitive-intel content for messaging, not strict lead attribution):**
1. Best-effort per-company attribution when a review's free text happens to name the reviewer's employer - flows into the normal per-company signal pipeline like Branch A/B.
2. Aggregated per-competitor pain-point corpus (the primary value) - common themes and representative "reason for switching" quotes, saved as reusable reference content that feeds Phase 8's Gemini outreach generation directly (real customer language instead of generic messaging), independent of whether any specific company was identified.
**Talking point:** "G2 reviews are anonymous by design, so I couldn't force this into the same per-company attribution model as the funding and hiring signals. Instead of treating that as a dead end, I reframed it around what G2 data is actually good for - real competitive-intelligence content for messaging - which is arguably a more honest use of the data than trying to over-attribute it."
**Correction (2026-07-10):** A deeper documentation pass (full parameter table + example JSON, not just a prose summary) before the first live run caught real field-name bugs: the input param is `productUrls` (full G2 URLs), not `productSlugs`; output fields are `reviewId`/`reviewText`/`nps`/`switchedFromOtherProduct`/`switchedReason`/`country` etc., not the invented names originally used. Fixed in `python/intent_g2.py` before any money was spent on a live run - see `docs/ISSUES.md`.
**Correction #2 (2026-07-10, after the user's first live run):** the real 60-review dataset revealed `switchedFromOtherProduct` is a `"yes"`/`"no"`/`"unknown"` flag, not a product name - and a truthy-string bug (`bool("no") == True` in Python) meant 42 of 60 reviews were being incorrectly flagged as switch signals instead of the real 13. Fixed with an explicit `== "yes"` check; corrected output re-validated against the real dataset. See `docs/ISSUES.md`.
**Status:** Decided.

## ADR-008: SEC EDGAR Form D filings replace Crunchbase for the funding signal
**Decision:** Branch A (funding/timing signal) pulls from the SEC's EDGAR full-text search API for Form D filings, not the Crunchbase API.
**What changed:** User's Crunchbase account is Basic tier only, which has no API access (Crunchbase removed its free API tier — as of 2026 it's Enterprise-only, ~$588+/year).
**Why SEC EDGAR:** Form D is a mandatory SEC filing any US company makes when raising private capital (seed through late-stage) — it's a free, public, government database with **no API key or signup required at all**. It includes issuer name, total offering amount, and date of first sale in real time. It does not include an explicit "Series A/B" label the way Crunchbase does, so we approximate funding stage from the offering amount (e.g., roughly $2M–$15M ≈ seed/Series A range, $15M–$50M ≈ Series B range) — a heuristic, documented as such, not a fabrication.
**Talking point:** "Crunchbase's API went Enterprise-only in 2026, so I went to the primary source it's partly derived from anyway — SEC Form D filings directly from EDGAR, free and public. I had to approximate funding stage from offering amount since Form D doesn't label rounds, which is a real tradeoff I'd flag to a GTM team relying on this data."
**Implementation detail (confirmed against SEC's own API docs, 2026-07-09):** Two-step flow, not one call. (1) Query `efts.sec.gov/LATEST/search-index` full-text search with `forms=D` and a date range → candidate companies that filed. (2) For each hit, fetch that filing's `primary_doc.xml` via CIK + accession number → extract `TotalOfferingAmount` and `IndustryGroup`. Requires a descriptive `User-Agent` header (`EDGAR_USER_AGENT` in `.env`) or EDGAR returns 403; rate limit 10 req/sec, no API key.
**Known gap:** EDGAR returns company legal name, not a website domain, but our Postgres schema dedupes on `domain`. Resolved by deferring domain lookup to the Clay enrichment step (Phase 7) rather than adding another tool just for this.
**Status:** Decided.

## ADR-010: Branch B rebuilt on Adzuna + Greenhouse/Lever, LinkedIn actor dropped entirely
**Decision:** Replace the Apify LinkedIn Jobs Scraper with a two-layer, scraping-free design: (1) Adzuna's official free job-search API for broad discovery of companies currently hiring PMs; (2) direct queries to each candidate company's own Greenhouse/Lever public job-board API (both official, free, no-auth, read-only) for the full authoritative posting list. `python/hiring_apify.py` and its tests are deleted, not kept as a dead fallback.

**Confirmed rate limits:** Adzuna free tier - 25 req/min, 250/day, 1,000/week, 2,500/month. **Known limitation:** Greenhouse/Lever are per-company, not searchable broadly - each candidate company's board token/client name is guessed from its name (e.g. "Acme Corp" → try `acme`, `acmecorp`, `acme-corp`), which won't always succeed. **Measured hit rate (2026-07-09 live run, 28 companies from a 30-day Adzuna "Product Manager" search): ~7% (2/28) resolved via Greenhouse/Lever.** Low, but the graceful-fallback design (falls back to Adzuna-only data, not an error) means this doesn't cost data - it just adds less value than hoped for most companies. **Talking point:** "After digging into the LinkedIn actor's own docs, avoiding personal login wasn't enough to fully remove the ToS risk of scraping LinkedIn at all. I rebuilt the branch on three officially-sanctioned free APIs instead - one for broad discovery, two for per-company verification - which is a stronger position than mitigating a scraping risk I didn't need to take in the first place."
**Status:** Decided.
