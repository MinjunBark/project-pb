> Owns: every notable tool/architecture choice, the alternatives considered, and why. This is the "why this tool and not another" material for interview prep — keep it accurate as decisions change, don't leave superseded entries in place.

## ADR-012: Branch C built on the Apify G2 scraper, purpose reframed as competitive-intel content
**Decision:** Branch C (intent signal) uses the Apify actor `automation-lab/g2-scraper` to pull reviews of named competitors (Aha!, Jira Product Discovery, ProductPlan, Craft.io). Per ADR-009, Claude writes and mock-tests the code but does not run live Apify calls itself - the user tests live to manage their own Apify usage.
**Alternatives checked first (2026-07-09), in order:** G2's own official Developer API and MCP server - both confirmed vendor-scoped (ADR-011), can't query competitor products at all. Capterra - no viable self-serve official API. Reddit - genuinely free official API, but not empirically tested for real coverage of these 4 competitor names given time constraints. Hacker News - free/official, empirically tested live: too noisy for "Aha!"/"ProductPlan" (common English phrases, not the product) and too low-volume for "Craft.io"/"Jira Product Discovery" to be usable.
**Why this actor specifically:** no login bypass (public review pages only), no proxy required (unlike other G2 actors that explicitly recommend residential proxies to evade anti-bot measures), built-in polite delays, minimal PII (reviewer display name only). A conscious, documented scraping tradeoff made only after the official alternatives above were checked and ruled out - not a default choice.
**Structural limitation discovered, and how it reshaped the design:** this actor's reviewer fields are name, country, region, company-size segment, and industry code - **not the reviewer's company name**. G2 reviews are largely anonymous by design. This means most reviews can't be attached to a specific company the way Branch A/B signals can, which matters because the scoring model's "BOTH signal" bonus depends on one company hitting both TIMING and INTENT. The original blueprint actually anticipated this - its Phase 1 spec listed the field as "company_name (if mentioned in review)," i.e., best-effort, not structured.
**Resulting two-part design (reframed per user's own articulation of G2's value - competitive-intel content for messaging, not strict lead attribution):**
1. Best-effort per-company attribution when a review's free text happens to name the reviewer's employer - flows into the normal per-company signal pipeline like Branch A/B.
2. Aggregated per-competitor pain-point corpus (the primary value) - common themes and representative "reason for switching" quotes, saved as reusable reference content that feeds Phase 8's Gemini outreach generation directly (real customer language instead of generic messaging), independent of whether any specific company was identified.
**Talking point:** "G2 reviews are anonymous by design, so I couldn't force this into the same per-company attribution model as the funding and hiring signals. Instead of treating that as a dead end, I reframed it around what G2 data is actually good for - real competitive-intelligence content for messaging - which is arguably a more honest use of the data than trying to over-attribute it."
**Correction (2026-07-10):** A deeper documentation pass (full parameter table + example JSON, not just a prose summary) before the first live run caught real field-name bugs: the input param is `productUrls` (full G2 URLs), not `productSlugs`; output fields are `reviewId`/`reviewText`/`nps`/`switchedFromOtherProduct`/`switchedReason`/`country` etc., not the invented names originally used. Fixed in `python/intent_g2.py` before any money was spent on a live run - see `docs/ISSUES.md`.
**Correction #2 (2026-07-10, after the user's first live run):** the real 60-review dataset revealed `switchedFromOtherProduct` is a `"yes"`/`"no"`/`"unknown"` flag, not a product name - and a truthy-string bug (`bool("no") == True` in Python) meant 42 of 60 reviews were being incorrectly flagged as switch signals instead of the real 13. Fixed with an explicit `== "yes"` check; corrected output re-validated against the real dataset. See `docs/ISSUES.md`.
**Status:** Decided.

## ADR-001: n8n as the sole orchestrator
**Decision:** Build the entire workflow in n8n. Do not build in Zapier or Salesforce Agentforce.
**Alternatives considered:**
- Zapier — weaker parallel branching, no merge-on-all-complete, gets expensive at this step volume, harder to run custom code steps.
- Salesforce Agentforce — not an orchestration/iPaaS tool; it's a framework for agents living natively inside a Salesforce org. We have no Salesforce org (HubSpot is the built CRM), so there's nothing to attach it to.
**Why n8n:** Full control over branching/data flow, self-hostable, handles HTTP nodes for every API in this stack, and is explicitly named in the actual job description.
**Status:** Decided.

## ADR-003: AWS dropped to narrative-only (reversed 2026-07-08)
**Original decision:** Raw signals land in S3 before merge/dedupe; the ICP scoring endpoint deploys to AWS Lambda via Mangum + API Gateway.
**What changed:** AWS requires completing billing/payment registration before IAM access is usable, even for free-tier usage. Given the user's standing preference to avoid providing card details for this portfolio project, we reversed course.
**Decision now:** AWS is not built against at all — same treatment as Salesforce/Marketo (ADR-005). Raw signal "landing" happens locally (written to `data/` / a Postgres staging table instead of S3). The ICP scoring endpoint (`api/main.py`) runs as a locally-hosted FastAPI service (`uvicorn`), not Lambda — this still gives genuine hands-on REST API experience (the JD's "Python, SQL, REST" line), just self-hosted instead of serverless.
**Alternatives considered:** LocalStack (Docker-based AWS emulator, no account/card needed) — would have preserved real boto3 S3/Lambda-shaped code without billing. Not chosen because the marginal interview value (one more line item) wasn't worth the added tool/time given everything else already in scope (Clay, Postgres, n8n, HubSpot, SEC EDGAR).
**Talking point if asked about AWS:** "I scoped AWS (S3 landing zone + Lambda-hosted scoring API) into the design, but given this is a portfolio project I chose not to attach billing to a personal AWS account — I built the same scoring API as a local FastAPI service instead, which is a one-step deploy to Lambda via Mangum if it needed to move to production."
**Status:** Decided (reversed from original).

## ADR-004: Postgres replaces the flat-file dedup store — hosted on Supabase (reversed 2026-07-09 from local Docker)
**Decision:** `seen_companies.json` is replaced by a real Postgres database (`companies`, `signals`, `leads` tables — see `sql/schema.sql`, plus `competitor_intel` for Branch C). Originally run locally via `docker-compose.yml`; **now hosted on Supabase** instead.
**Original alternatives considered (2026-07-08):** AWS RDS — more "real" cloud footprint to reference, but costs money/time and needs VPC/security group setup for no added learning value in a 5-day build. Local Docker was chosen over RDS at that point.
**Why reversed to Supabase:** Supabase is literally Postgres under the hood (not a different database), hosted for free with **no credit card required** (confirmed 2026-07-09, unlike AWS) — restores a genuine hosted-cloud talking point that AWS would have given, without AWS's billing wall. Bonus: a web table editor to visually show data during the interview instead of only `psql` output. Zero code changes needed — same `sql/schema.sql`, same `DATABASE_URL`-based connection, just pointed at a hosted instance instead of `localhost`.
**Known tradeoff:** free Supabase projects pause after 7 days of inactivity and need a manual dashboard login to unpause — not a blocker, just something to do before the interview if the project's gone quiet.
**Connection detail (found live, 2026-07-09):** Supabase's "Direct connection" string resolves over IPv6 by default, which failed to resolve at all on this network (DNS lookup failure). Fixed by using the **Session pooler** connection string instead (`aws-0-<region>.pooler.supabase.com`), built for IPv4 networks — connected successfully to a live PostgreSQL 17.6 instance. See `docs/ISSUES.md` for the full debugging note.
**Status:** Decided and live-verified. `docker-compose.yml` removed (superseded, not kept as unused fallback).

## ADR-005: HubSpot is the only built CRM; Salesforce/Marketo are narrative-only
**Decision:** All real CRM writes go to a HubSpot account. Salesforce and Marketo are not built against — they're prepared talking points about how the same architecture would map to them (swap REST auth + object model).
**Why:** Could not confirm Productboard's actual internal CRM/MAP stack (their own product integrates with both HubSpot and Salesforce for customers, which says nothing about their internal GTM stack). The JD's "Salesforce, Marketo, HubSpot, etc." phrasing reads as generic "familiarity with concepts," not a confirmed stack reveal. HubSpot has the fastest free account to stand up in a 5-day window.
**Correction (2026-07-08):** Original guidance said "HubSpot developer sandbox," inherited from the initial blueprint's imprecise wording. Confirmed via HubSpot's own docs: **Private Apps (what we need for the API token) are not available in Developer accounts at all** — only in a regular HubSpot CRM account with Super Admin permissions. User's existing developer account (used for building/publishing marketplace apps, with "Legacy Apps"/"Projects"/"Development" nav) is unrelated and unused; a separate plain free HubSpot CRM signup (hubspot.com, not developers.hubspot.com) is what actually hosts the Private App token.
**Status:** Decided.

## ADR-007: Claude authors Python logic; user builds the n8n workflow itself
**Decision:** Claude Code writes and pytest-covers the underlying logic as standalone Python (scoring, classification, outreach generation, the locally-hosted scoring API). The user builds the actual n8n workflow — the HTTP Request nodes, Merge/Filter nodes, branching — by hand in the n8n UI, with Claude guiding node-by-node. Claude does not author `n8n/workflow.json` from scratch or push workflows via n8n's REST API.
**Alternatives considered:** Claude authors the full workflow JSON and pushes it to n8n via its REST API (n8n does expose one — `POST /workflows` etc. — no dedicated n8n MCP server was available in this session, but the plain REST API would have worked fine). Faster and fully version-controlled, but means Claude built the orchestration too.
**Why:** n8n is explicitly named in the JD and is exactly the kind of tool Darrell will probe hands-on ("walk me through the branching logic," "how'd you handle rate limits in that HTTP node"). Having Claude author the workflow blind would produce a working pipeline but not real n8n fluency — defeating the point of this build as learning, not just output. `n8n/workflow.json` in the repo is retained only as an optional exported snapshot of what the user built, for version history.
**Status:** Decided.

## ADR-009: Claude doesn't live-test Apify actors; user tests manually
**Decision:** For any Apify-based branch (Branch B / LinkedIn jobs, Branch C / G2 reviews), Claude writes the code against the actor's documented input/output schema but does not run live calls against it. The user runs the first live test themselves, in the Apify console or locally with their own token.
**Why:** Unlike SEC EDGAR (free, unlimited, no auth), Apify actor runs consume the user's account credits/compute units. EDGAR got a live Claude-run smoke test during Branch A specifically because it was free to do so; Apify does not get the same treatment by default.
**Status:** Decided.

**Addendum (2026-07-09) — actor selection + safety review, prompted by user question:**
`curious_coder~linkedin-jobs-scraper` was inherited from the original `gtm-signal-blueprint.md` spec, not independently benchmarked against alternative LinkedIn actors on Apify's marketplace. Claude verified its I/O schema before writing code, but had not checked its safe-usage guidance until asked directly. Follow-up found: the actor scrapes LinkedIn's public/incognito-accessible job search and does **not** require the user's personal LinkedIn login/cookies — meaningful, since actors that do use a real session risk that personal account getting flagged/banned by LinkedIn's bot detection. However: (1) the actor's own README has no explicit rate-limit/concurrency guidance, and (2) LinkedIn's ToS still generally prohibits automated scraping regardless of public-page access — the same category of legal-gray-area risk already accepted for G2 scraping (and part of why Reddit scraping was dropped earlier, ADR-006). Given occasional, modest-volume use here (not high-frequency production scraping), residual risk is low but not zero.
**Talking point:** "I picked this actor because it avoids the biggest personal risk — it doesn't touch your own LinkedIn login — but I was upfront with myself that the underlying ToS risk of scraping at all doesn't disappear just because the actor is careful about account safety. That's a real tradeoff I'd flag to a GTM team, not something to gloss over."

## ADR-008: SEC EDGAR Form D filings replace Crunchbase for the funding signal
**Decision:** Branch A (funding/timing signal) pulls from the SEC's EDGAR full-text search API for Form D filings, not the Crunchbase API.
**What changed:** User's Crunchbase account is Basic tier only, which has no API access (Crunchbase removed its free API tier — as of 2026 it's Enterprise-only, ~$588+/year).
**Why SEC EDGAR:** Form D is a mandatory SEC filing any US company makes when raising private capital (seed through late-stage) — it's a free, public, government database with **no API key or signup required at all**. It includes issuer name, total offering amount, and date of first sale in real time. It does not include an explicit "Series A/B" label the way Crunchbase does, so we approximate funding stage from the offering amount (e.g., roughly $2M–$15M ≈ seed/Series A range, $15M–$50M ≈ Series B range) — a heuristic, documented as such, not a fabrication.
**Talking point:** "Crunchbase's API went Enterprise-only in 2026, so I went to the primary source it's partly derived from anyway — SEC Form D filings directly from EDGAR, free and public. I had to approximate funding stage from offering amount since Form D doesn't label rounds, which is a real tradeoff I'd flag to a GTM team relying on this data."
**Implementation detail (confirmed against SEC's own API docs, 2026-07-09):** Two-step flow, not one call. (1) Query `efts.sec.gov/LATEST/search-index` full-text search with `forms=D` and a date range → candidate companies that filed. (2) For each hit, fetch that filing's `primary_doc.xml` via CIK + accession number → extract `TotalOfferingAmount` and `IndustryGroup`. Requires a descriptive `User-Agent` header (`EDGAR_USER_AGENT` in `.env`) or EDGAR returns 403; rate limit 10 req/sec, no API key.
**Known gap:** EDGAR returns company legal name, not a website domain, but our Postgres schema dedupes on `domain`. Resolved by deferring domain lookup to the Clay enrichment step (Phase 7) rather than adding another tool just for this.
**Status:** Decided.

## ADR-011: G2's official Developer API confirmed vendor-scoped, not usable for competitor review data
**Decision:** Do not use G2's official self-serve Developer Portal API for Branch C (intent signal). It cannot answer our actual question ("what are people saying about competitor products like Aha!, Jira PD, ProductPlan, Craft.io") because it only exposes products/reviews the calling account itself owns as a claimed G2 vendor listing.
**Why investigated:** Before defaulting to the Apify G2 scraper (which explicitly recommends residential proxies to evade G2's anti-bot measures - a more overt scraping posture than the LinkedIn actor had), checked whether G2 has a genuinely open official API first, same discipline applied to the Crunchbase→EDGAR and LinkedIn→Adzuna/Greenhouse/Lever pivots.
**How confirmed:** User signed up for G2's free Developer Portal, generated an access token scoped to Products/Reviews/Product Ratings (Read). The Products permission's own description read "list all products **owned by current account**." Empirically tested: `GET data.g2.com/api/v2/products` returned `{"data": [], "record_count": 0}` - confirming the account (which owns no G2 vendor listing) gets zero products back, proving the endpoint is ownership-scoped, not a general product/review directory. A separate "review syndication" API exists but requires a partner-issued key from a G2 representative - not self-serve, not applicable here either.
**Also checked: G2 MCP server** (documentation.g2.com/docs/g2-mcp-server) - same conclusion, more strongly stated. Explicitly scoped to "your own registered products" (buyer intent on your listing, competitors your prospects compare you against - not a way to query a competitor's product directly), and requires "an active G2 subscription" to even connect - not reachable on a free account at all, unlike the REST API tested above.
**Status:** Decided - G2's official API and MCP server are both not viable for this use case. User decided (2026-07-09) to proceed with the Apify G2 scraper as a conscious, documented tradeoff rather than defaulting to it - see ADR-012 for the specific actor and its safety posture.

## ADR-010: Branch B rebuilt on Adzuna + Greenhouse/Lever, LinkedIn actor dropped entirely
**Decision:** Replace the Apify LinkedIn Jobs Scraper with a two-layer, scraping-free design: (1) Adzuna's official free job-search API for broad discovery of companies currently hiring PMs; (2) direct queries to each candidate company's own Greenhouse/Lever public job-board API (both official, free, no-auth, read-only) for the full authoritative posting list. `python/hiring_apify.py` and its tests are deleted, not kept as a dead fallback.
**Why:** Prompted by the user questioning the LinkedIn actor's safety posture (ADR-009 addendum) and asking for genuinely free/official alternatives "similar to EDGAR." Research (independently corroborated by the user's own ChatGPT research) confirmed three real options - Adzuna (broad search, official API, free `app_id`/`app_key`), Greenhouse and Lever (official public read-only per-company job-board APIs, no auth at all). This fully removes the LinkedIn ToS/scraping risk rather than just mitigating it.
**Confirmed rate limits:** Adzuna free tier - 25 req/min, 250/day, 1,000/week, 2,500/month. Greenhouse and Lever publish no hard limit on their public read endpoints (the stricter numbers found for both platforms - Greenhouse's Harvest API at 50 req/10sec, Lever's 2 POST/sec - apply to different, authenticated write/application endpoints not used here).
**Known limitation:** Greenhouse/Lever are per-company, not searchable broadly - each candidate company's board token/client name is guessed from its name (e.g. "Acme Corp" → try `acme`, `acmecorp`, `acme-corp`), which won't always succeed. **Measured hit rate (2026-07-09 live run, 28 companies from a 30-day Adzuna "Product Manager" search): ~7% (2/28) resolved via Greenhouse/Lever.** Low, but the graceful-fallback design (falls back to Adzuna-only data, not an error) means this doesn't cost data - it just adds less value than hoped for most companies. A wrong guess could theoretically collide with a different company's board (low probability, not zero). Adzuna itself only returns `company.display_name`, not a domain - same domain-resolution gap already deferred to Clay for Branch A (ADR-008) now also applies here.
**Talking point:** "After digging into the LinkedIn actor's own docs, avoiding personal login wasn't enough to fully remove the ToS risk of scraping LinkedIn at all. I rebuilt the branch on three officially-sanctioned free APIs instead - one for broad discovery, two for per-company verification - which is a stronger position than mitigating a scraping risk I didn't need to take in the first place."
**Status:** Decided.
