**Crash-loop bug in `import_enriched_companies()`:** Clay's real waterfall enrichment mismatched 3 different companies (Chrome-extension-style Product Hunt launches) to the same placeholder domain (`google.com`). `companies.domain` is UNIQUE, so the 2nd/3rd row raised `psycopg2.errors.UniqueViolation` - uncaught, this killed the whole import loop every single time, and since the file was never moved out of `data/clay/incoming_domain/` on failure, the 60s poller retried the identical failing import forever (visible in the user's own Discord screenshot: the exact same error posted 3 times in a row). Fixed: catch `UniqueViolation` per-row, `conn.rollback()`, skip just that row and continue - matches the existing "nothing to write, not a stop-the-world error" pattern already used for a missing domain value. 1 new pytest case.

**User design pushback, correctly identified a real limitation:** the user asked why the system was "silently" auto-resuming and expected "upload it on the Discord channel" to mean a literal Discord attachment upload, not a local file drop. Root cause explained (a webhook can only post *out*, never read messages back - genuinely needs a bot with its own token), and the user explicitly chose to build the real bot rather than keep the local-folder-only convention.

## Redesign v2, Tier 6: real Discord bot for Clay enrichment upload (2026-07-13)
Built on top of Tier 5, not a replacement - the local-folder + 60s poller stays exactly as a fallback. New `discord.py==2.4.0` dependency.

**`python/enrichment.py`** gained `_detect_enrichment_kind(header)` (content-based, not filename-based - checks for the same `EMPLOYEE_COUNT_COLUMNS`/`INDUSTRY_COLUMNS` candidates `import_enriched_demographics()` already uses to recognize a demographics file, falls back to a `Domain`/`domain` column for a domain file, returns `None` if neither shape is recognized) and `save_incoming_enrichment_file(kind, filename, content)` (writes real bytes into the correct known-folder, microsecond-timestamp-prefixed so two uploads of the same original filename can never collide).

**New `utils/discord_bot.py`** - a real `discord.Client` watching a single configured channel (`DISCORD_CLAY_CHANNEL_ID`) for CSV attachments. All real logic lives in `handle_upload()`, a plain async function with no `discord.py` types in its signature (testable without mocking `Client`/`Message`/`Attachment`): detects the kind, saves it, replies immediately, then calls `resume_after_enrichment()` right away (not waiting for the next 60s poll tick) and posts a second reply once that finishes. `create_bot_task()` returns `None` (does not raise) if `DISCORD_BOT_TOKEN`/`DISCORD_CLAY_CHANNEL_ID` aren't set - the bot is a pure enhancement, missing credentials can't block FastAPI startup. Wired into the same `@app.on_event("startup")` handler the Tier 5 poller already uses - one process, no new bot infrastructure to run separately.

**Real, live-hit naming collision found and fixed during setup:** `utils/discord.py` (this project's own webhook helper, written in Phase 9) has the exact same importable name as the pip-installed `discord.py` library. Since every entry point inserts `utils/` at the front of `sys.path`, `import discord` inside the new bot module resolved to *our own* module instead of the real SDK - crashed FastAPI startup with `AttributeError: module 'discord' has no attribute 'Intents'`. Fixed by renaming the file to `utils/discord_webhooks.py` and updating the 4 real call sites (`api/main.py`, `python/full_pipeline_run.py`, `python/pipeline.py`, `tests/test_discord.py` -> `tests/test_discord_webhooks.py`) to `import discord_webhooks as discord` - every existing `discord.send_*()`/`discord.build_*()` call site is unchanged.

**Also hit and fixed a real local-machine issue, not a code bug:** the bot's first connection attempt failed with `SSLCertVerificationError` - this Python.org installation had never run its bundled `Install Certificates.command`, so `aiohttp` (used by `discord.py`) couldn't verify Discord's TLS certificate. Ran the official fix script; resolved immediately.

**6 new pytest cases** (`test_discord_bot.py`, new file - no `pytest-asyncio` dependency added; each test runs the async `handle_upload()` via a plain `asyncio.run()` call from an ordinary sync test function). **281/281 tests passing.**

**Live-verified end-to-end with a real human upload, not a simulated one.** Restarted `uvicorn`, confirmed via the bot's own REST identity check (`GET /users/@me`) that the token is valid and the bot is real. The user then dragged their own real, already Clay-enriched domain CSV directly into `#clay-enrichment` as a genuine Discord attachment (not a webhook post, not a local file drop). Confirmed via the bot's own message history (fetched over the REST API, since this session has no direct Discord UI access): the bot replied "✅ Got it..." within under a second of the upload, then correctly took ~5.5 minutes for a real leadership-check pass across all 112 domain-having companies (up from 22, since the domains had just been backfilled) before posting "🔄 Pipeline auto-resumed: 0 qualified lead(s) out of 114 companies evaluated" - matching the real timing of an earlier equivalent poller-triggered run almost exactly (~6 min for the same real workload). Confirmed the SDR digest posted to `#sdr-digest` at the same moment with a fresh date-stamped embed. Zero manual file handling, zero re-trigger - a real end-to-end proof of the feature the user asked for.

## `include_demographics_enrichment` toggle + real diagnosis: identical branch counts + 0 qualified leads (2026-07-13)
After Tier 6, the user asked to run a full test excluding Clay demographics enrichment (focusing only on the domain-enrichment human-in-the-loop path). Added `include_demographics_enrichment: bool = True` to `run_full_cycle()`/`RunFullCycleRequest`, mirroring the existing `include_branch_c` opt-out pattern - the request half of `_run_clay_pickup_and_request_phase()` skips the demographics queue and logs an honest `⏭️` skip (pickup still always runs, since picking up already-completed work is harmless). 2 new pytest cases, **283/283 passing**. Live-verified: real full-cycle run posted only a domain request to `#clay-enrichment`, with `clay_requests.demographics` recorded as `{"skipped": true, "reason": "excluded for this run"}` in the landed run log.

The user then pushed back hard on two real observations from that live run: identical branch signal counts across 5+ runs, and 0 qualified leads out of 134 companies every time - suspecting a scraping bug rather than accepting either at face value. Full investigation and real fix documented in `docs/ISSUES.md`'s "[Not a bug] Branch A/B/D returning identical..." and "[Redesign v2] Diagnosed why 0 qualified leads keeps happening..." entries. Short version: identical branch counts is genuinely correct behavior (fixed lookback windows re-queried within hours of the last run against static real external data - proven with a fresh live EDGAR call at the moment of investigation, not just old logs). 0 qualified leads led to a real, previously-undiagnosed root cause three layers upstream of scoring: `hiring_adzuna.py`'s bare `"Product Manager"`/`"Product Operations"` keyword search had zero industry filtering, letting real staffing/recruiting agencies (11 of 33 unmatched companies) and non-SaaS megacorps through - agencies that can never represent real buying intent no matter how good the downstream ATS-slug-matching heuristic is. Added `_is_staffing_agency()` (regex + curated known-agency-name list, same documented-heuristic spirit as `funding_edgar.py`'s `FUND_NAME_PATTERN`) to `hiring_adzuna.py`, wired into `normalize_adzuna_results()`. 4 new pytest cases, **287/287 passing**. Live-verified: a real Adzuna call before the fix returned 35 companies including all 11 known agencies; the identical live call after the fix returned 24 companies with zero agencies present. Explicitly did not attempt a "megacorp" filter (no real firmographic signal exists this early in the pipeline) or lower the 70-point threshold (already rejected earlier this session as fabricating quality) - documented as a real, remaining gap rather than silently declared "fixed."
