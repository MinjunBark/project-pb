# Issues & Debugging Log

> Owns: bugs, debugging dead-ends, and how each was resolved. This is the "what broke and how'd you fix it" material for the interview — log the real dead-end, not just the final fix, since the dead-end is often the more interesting part to talk about.

## [Redesign v2] Product Hunt: copying the OAuth Client Secret instead of the Developer Token
**Symptom:** First live call to `producthunt_launches.search_recent_launches()` returned a real `401 invalid_oauth_token` - "Please supply a valid access token."
**Root cause:** Product Hunt's app-creation page shows a Client ID/Client Secret pair (for the full OAuth authorization-code flow) prominently, but a separate, explicit **"Developer Token"** button further down the same app dashboard page is what actually works as a direct Bearer token for a server-side script. The user had copied the Client Secret, not the Developer Token - both are long opaque strings, easy to confuse from the UI alone.
**Fix:** Regenerated/located the actual Developer Token from the app dashboard, confirmed live via a minimal `{ viewer { user { name } } }` query (returned the real account name) before re-testing the full launches query.
**Also worth noting:** the app-creation form itself required an HTTPS redirect URI even though the Developer Token path never uses it - `https://example.com/callback` (a non-functional placeholder) was accepted, since Product Hunt only validates the URI's scheme at creation time, not that it's live/reachable.
**Talking point:** "A 401 with a real, structured error body (not a silent failure) let me diagnose this in one request instead of guessing - `invalid_oauth_token` immediately ruled out 'wrong endpoint' or 'malformed header' and pointed straight at 'wrong credential type,' which matched what I found once I re-checked the dashboard."

## [Phase 3] EDGAR returns foreign private issuers - no built-in US filter
**Symptom:** A live Branch A sample included "Pulsenmore Ltd. (PLSM)" based in Israel, alongside 7 real US companies.
**Root cause:** SEC EDGAR has no country filter - any company that legally files a Form D shows up, including foreign private issuers who raise US capital.
**Fix:** Added `is_us_location()` to `funding_edgar.py`, checking that `biz_location` ends in a valid 2-letter US state code (EDGAR shows foreign issuers as a bare country name, e.g. "Israel", with no state code). Filters at the search step, before the more expensive per-filing detail fetch. Live-verified: re-ran the same 60-day search, now returns 11 US-only companies, Pulsenmore correctly excluded.
**Talking point:** The original blueprint's ICP never specified geography - this was a gap the user caught by asking to verify real data against the ICP, not something either of us assumed correctly upfront.

## [Phase 2] Truthy-string bug: "no" and "unknown" incorrectly counted as switch signals
**Symptom:** After the user's first live G2 run (60 real reviews across all 4 competitors), inspecting the raw data showed `switchedFromOtherProduct` takes values `"yes"`, `"no"`, `"unknown"`, or `null` - it's a categorical flag, not (as assumed) the name of the product the reviewer switched from.
**Root cause:** `normalize_g2_reviews()` computed `is_switch_signal` as `bool(review.get("switchedFromOtherProduct") or review.get("switchedReason"))`. In Python, `bool("no")` and `bool("unknown")` are both `True` - any non-empty string is truthy. Real distribution across the 60 reviews: 13 `"yes"`, 25 `"no"`, 4 `"unknown"`, 18 `null`. The buggy version would have flagged 42 of 60 reviews as switch signals when only 13 actually were.
**Fix:** Changed the check to `review.get("switchedFromOtherProduct") == "yes"` specifically. Also renamed the misleadingly-named `prior_product` field (which held `"yes"`/`"no"`/`"unknown"`, not a product name) to `switched_from_other_product` (a proper boolean). The actual name of a prior product, when mentioned, lives in free text (`switch_reason`/`text`) - still a Phase 5 Gemini extraction job, not a structured field. Added a regression test (`test_normalize_does_not_flag_no_or_unknown_as_switch_signal`) using real-shaped data. Re-ran the corrected code against the actual 60-review dataset: exactly 13 switch signals now, matching the real "yes" count precisely.
**Talking point:** A classic "truthy string" bug - Python (and JS) treat any non-empty string as `True`, so a categorical field like this needs an explicit equality check, not a bare `bool()`. Real downloaded data caught it; the mocked tests alone wouldn't have, since the mocks were written with the same wrong assumption baked in.

## [Phase 2] G2 actor input param and output field names were wrong before first live run
**Symptom:** Before running the first live Apify test, user asked for a more thorough documentation pass on `automation-lab/g2-scraper` than the initial research. That deeper pass (the actor's full parameter table and example JSON, not just a summary) revealed the original `python/intent_g2.py` was built against wrong names throughout: input param `productSlugs` should have been `productUrls` (an array of full G2 URLs, not bare slugs), and output fields like `id`, `text`, `npsScore`, `priorProductName`, `reasonForSwitching`, `reviewerCountry` should have been `reviewId`, `reviewText`, `nps`, `switchedFromOtherProduct`, `switchedReason`, `country`.
**Root cause:** The initial research pass (before writing the code) summarized the actor's fields in prose rather than pulling the actual parameter table and example JSON output - close enough to sound plausible, wrong in the specifics.
**Fix:** Re-fetched the actor's full documentation (console readme + public store page + targeted search for the input-schema page) to get the literal parameter table and example JSON, then corrected every field mapping in `normalize_g2_reviews()` and the input body in `run_g2_review_scraper()`. Added `build_product_review_url()` to construct the full URL the actor actually expects. All 6 tests (5 existing + 1 new) updated to match and passing.
**Talking point:** This would have been a genuinely embarrassing live-run failure (or worse, silently wrong data - e.g. `None` for every field if the wrong keys just don't match) had it not been caught before spending real Apify credits. Good argument for always pulling the literal parameter table and example payload, not a prose summary, before writing integration code - especially before a paid live run.

## [Phase 3 prep] Supabase direct connection failed DNS resolution
**Symptom:** `psycopg2.connect()` against Supabase's "Direct connection" URI (`db.<ref>.supabase.co`) failed with `could not translate host name ... nodename nor servname provided`.
**Root cause:** Supabase's direct-connection host resolves over IPv6 by default; the local network couldn't resolve/route it at all.
**Fix:** Switched to Supabase's "Session pooler" connection string instead (`aws-0-<region>.pooler.supabase.com`), built specifically for IPv4 networks. Connected successfully — confirmed live against a real PostgreSQL 17.6 instance.
**Talking point:** A reminder that "free and hosted" doesn't mean "zero network friction" — IPv6-only defaults are a real, common gotcha, and knowing the pooler fallback existed (rather than assuming the direct string was the only option) is exactly the kind of infra troubleshooting this role expects.

## [Phase 1] EDGAR Form D search returns mostly investment funds, not operating companies
**Symptom:** A raw date-range query (`forms=D`, no search text) against EDGAR's full-text search API returned 10,000+ hits, and manually inspecting the first page showed almost entirely venture fund/SPV entities raising their own capital ("1EP Ventures I, L.P.", "OT YC Fund IV, LLC", "GTOWN CENTURY LP, LLC") rather than operating startups that just received funding.
**Root cause:** Form D is filed by whoever is the *issuer* of the private securities — for a venture fund raising money from its own LPs, the fund itself is the issuer, so it files a Form D exactly like an operating company raising a Series B would. Nothing in a plain date-range query distinguishes the two.
**Fix:** Added a required search keyword (e.g. "software", "SaaS", "platform") to the full-text query — narrowed 10,000+ hits to 13 for a similar window, and surfaced real operating companies ("Kepler Software, Inc.", "Blacksmith Software Inc.", "Nova AI Software Inc."). Also added a name-based heuristic filter (excludes names containing "fund", ending in "LP"/"L.P.", or containing "SPV") as a second safety net. Confirmed via live test calls to `efts.sec.gov` before writing any parsing code, rather than assuming the schema from secondhand descriptions.
**Talking point:** "The naive version of this query would've flooded the pipeline with venture funds instead of prospects. I caught it by actually inspecting live API output before writing code, not by trusting the docs blindly — same instinct as validating any third-party data source before building on it."

## [Phase 5] GEMINI_API_KEY_2 is not a valid key, and gemini-2.0-flash is deprecated (not actually a quota problem)
**Symptom:** Live-testing `python/classify.py` against real G2 review text: `GEMINI_API_KEY_2` produced an immediate 400 (Bad Request) - inspecting `.env` (without printing the actual value) showed it's only 16 characters versus the working `GEMINI_API_KEY`'s 53, with `# Dual-key fallback for rate limits` apparently concatenated onto the value rather than treated as a separate comment. Removing it and retrying with just the primary key against the originally-hardcoded model, `gemini-2.0-flash`, returned a 429 that looked like quota exhaustion: `"You exceeded your current quota, please check your plan and billing details."`
**Root cause (corrected):** the user flagged that this specific error, on this specific model, is a known symptom of using a deprecated model, not real usage-based quota exhaustion - confirmed by fetching `ai.google.dev/gemini-api/docs/models`, which lists `gemini-2.0-flash` as deprecated/shut down. Querying `GET /v1beta/models` directly against the real key confirmed a completely different current model lineup (Gemini 2.5 and 3.x series). Trying the "obvious" successor, `gemini-2.5-flash-lite`, returned a **404** - `"This model ... is no longer available to new users"` - so even a seemingly-current pinned model name can already be stale. `GEMINI_API_KEY_2` being invalid is a separate, still-real environment misconfiguration (never properly filled in).
**Fix:** Changed `utils/gemini.py`'s `DEFAULT_MODEL` from `gemini-2.0-flash` to `gemini-flash-lite-latest` - a rolling alias Google maintains to always point at the current lite model, chosen specifically so this doesn't go stale the same way a pinned version number just did twice in one debugging session. Live-verified end-to-end against real G2 review text (15 real reviews, 0 attributed - expected per ADR-012's anonymity finding) and against synthetic text that does name a company ("We use this at Acme Corp..." correctly extracted "Acme Corp"; a second call in the same burst hit a real, transient 429 - genuine free-tier RPM rate limiting this time, confirmed by succeeding again 15 seconds later). `GEMINI_API_KEY_2` is still invalid and needs the user to supply a real second key - not blocking, since the primary key now works, but the dual-key fallback has no real second key to fall back to yet.
**Talking point:** "My first live test looked like a quota problem and I almost treated it as one - but the user caught that it was actually a deprecated-model symptom, and even the 'obvious' next model turned out to already be retired for new accounts. I ended up querying the live model list directly instead of trusting any single doc page, and landed on a rolling alias specifically so a pinned version number can't quietly go stale on me again."

## [Phase 6] load_latest_raw_signals() picked a stale file because it sorted filenames as strings, not by actual time
**Symptom:** Live-testing the full merge/enrichment flow: landed a fresh 22-company Branch A run (`branch_a_20260711T022934Z.json`), then called `merge_signals.run_full_merge()` expecting it to use that fresh data. It silently loaded an old 8-company file instead (`branch_a_funding_20260710T004132Z.json`, a leftover from earlier ad-hoc testing) - including "Pulsenmore Ltd." (the Israeli filer `is_us_location()` was specifically built to exclude), which had already been caught and deleted once from the live database via an earlier live test, then reappeared.
**Root cause:** `raw_landing.py`'s `load_latest_raw_signals()` picked "latest" via `sorted(glob.glob(pattern))[-1]` - a plain string sort on filenames. The stale file's name (`branch_a_funding_...`) starts with `f` right after the shared `branch_a_` prefix; the fresh file's name (`branch_a_...`) continues straight into a digit (`2`). ASCII-wise, `'f' > '2'`, so the OLDER file sorted after the NEWER one and got treated as "most recent" - a filename-shape collision the function had no way to detect, since it never looked at actual file timestamps at all.
**Fix:** Changed `load_latest_raw_signals()` to select by `os.path.getmtime()` (real file modification time) instead of filename string order. Added a regression test (`test_load_latest_raw_signals_uses_file_mtime_not_filename_string_sort`) that reconstructs this exact scenario - a differently-prefixed older file that would out-sort a genuinely newer one alphabetically - and confirms mtime-based selection picks correctly. Deleted the stale file from `data/raw/` and the reintroduced Pulsenmore row from Supabase (twice, since the bug reintroduced it mid-debugging before the fix landed) via a scoped `DELETE ... WHERE name LIKE 'Pulsenmore%'` (a blanket `DELETE FROM companies` with no WHERE clause was correctly blocked by the environment's safety controls when attempted as a shortcut).
**Also live-verified as a result:** `merge_signals.run_full_merge()` (previously never run against real Supabase data - flagged as an open item in `docs/HANDOFF.md`) now confirmed working end-to-end: 22 real companies merged correctly, all with valid US `biz_location` values. `python/enrichment.py`'s `export_companies_needing_domain()` also confirmed live - produced a correct 22-row CSV from the real database, ready for Clay import.
**Talking point:** "A live test surfaced a bug that mocked tests never would have - two files that individually looked fine both matched the same glob pattern, and my sort assumed filenames would always share one naming convention. Fixing it to sort by actual file time instead of the filename string is the more correct approach regardless, not just a patch for this one collision."

## [Not a bug] A live Branch B run + full re-score found zero real qualifying companies
**Symptom:** Ran Branch B live for the first time against real data (39 real companies found via Adzuna/Greenhouse/Lever), merged into Supabase alongside the existing 22 real EDGAR companies (61 total, confirmed zero name overlap between the two branches - `companies_from_funding: 22 + companies_from_hiring: 39 = distinct_company_ids: 61` exactly), then re-scored all 61 real companies. **0 qualified (score >= 70).** Highest real score: 25 (funding-only companies, `funding_recency` alone).
**Root cause:** Not a bug - this is the scoring model's ceiling working exactly as designed. TIMING-only companies cap at 50 points max (funding_recency 25 + pm_posting_count 15 + product_ops_posting bonus 10), and the BOTH-signal +10 bonus only applies when a company ALREADY has both TIMING and INTENT points - so a TIMING-only company can never cross 70 on timing alone, by design. Today's real snapshot found genuinely zero companies with both a real funding event AND a real PM-hiring signal (Branch A and Branch B are independent broad searches, not targeted at each other), and zero of the 39 real job postings mentioned any of the 4 tracked competitors, so the INTENT bucket stayed at 0 for every company.
**Why this is correct, not broken:** the entire premise of "high-intent" scoring (vs. a static demographic list) is requiring multiple independent signals to overlap on the same real company before it's worth prioritizing. A single day's snapshot legitimately producing zero qualified leads is a real, expected outcome for a strict multi-signal model - not evidence the pipeline is broken. The earlier live pipeline test (with one company and a synthetic supporting signal) already proved the qualify → dedupe → outreach → write path works correctly when a lead DOES cross the threshold; this run proves the equally important other half - that non-qualifying companies are correctly and silently discarded, with zero wasted Gemini calls or HubSpot writes.
**What would change this in a real production run:** (1) accumulating signal history over multiple days (the blueprint's original daily-cron design) - a company funded today might post a PM role next week, and only repeated runs let that overlap actually occur; (2) backfilling `employee_count`/`is_saas` via a second Clay enrichment pass (currently only `domain` was backfilled) - closes some of the demographic bucket's up-to-25-point gap, though still not enough alone to cross 70 without a second real signal; (3) widening Branch B's keyword/lookback scope to increase the odds of hitting the same real companies Branch A already found.
**Talking point:** "Running the two live branches back-to-back and getting zero overlap wasn't a failure - it's exactly the kind of thing you only learn by running the real system instead of assuming it'll obviously find something. It also confirmed the negative path works: 61 real companies went through scoring, all correctly got discarded below threshold, and nothing wasteful happened downstream - no wasted API calls, no junk records in HubSpot. That's arguably more validating than one test lead succeeding."

## [Phase 9/n8n prep] api/main.py never loaded .env, so every real endpoint failed under a standalone uvicorn process
**Symptom:** Extended `api/main.py` with n8n-facing endpoints (`/branch-a/run`, etc.) and live-tested by starting a fresh `uvicorn` process (not reusing any Python session that had already run `load_dotenv()`). `POST /branch-a/run` returned a `500 Internal Server Error`.
**Root cause:** `api/main.py` never called `load_dotenv()` - every other script in this entire session explicitly loaded `.env` before touching any credential-reading function, but the FastAPI app was the one place that assumed environment variables would already be present. Confirmed via the real traceback: `funding_edgar._headers()` raised its own correctly-designed `RuntimeError` (`EDGAR_USER_AGENT must be set in .env`) - the guard worked exactly as intended, it just proved the real gap was one level up.
**Fix:** Added `load_dotenv()` at the top of `api/main.py`, before the `sys.path` inserts, pointing explicitly at the repo's `.env` file (`Path(__file__).resolve().parent.parent / ".env"`) so it works regardless of the working directory `uvicorn` is launched from - important since n8n itself may invoke this service from a different working directory than a developer's terminal.
**Live-verified after the fix:** re-started `uvicorn` fresh and confirmed all four testable endpoints work for real - `/branch-a/run` (22 real companies), `/branch-b/run` (40 real companies, fresh live Adzuna/Greenhouse/Lever run), `/merge/run` (62 distinct companies merged), `/pipeline/run-all` (62 companies evaluated, all correctly `not_qualified`, consistent with the earlier finding). `/branch-c/run`'s wiring is confirmed only via its mocked unit test - not live-tested, since each real call costs Apify money (ADR-009).
**Talking point:** "This is exactly the kind of bug that only shows up when you actually run the service the way it'll really be run - a fresh process, not a warm Python session that already had environment variables loaded from an earlier script. Testing via `uvicorn api.main:app` for real, not just FastAPI's TestClient in pytest, caught it immediately."

## [Redesign v2, Tier 5] import_enriched_companies() crash-looped forever on a real Clay waterfall mismatch
**Symptom:** The user's live-verification screenshot of `#ops-progress` showed the identical error posted 3 times in a row: `Clay enrichment auto-resume poll failed: duplicate key value violates unique constraint "companies_domain_key" DETAIL: Key (domain)=(google.com) already exists.`
**Root cause:** Clay's real waterfall enrichment mismatched 3 different companies (all Chrome-extension-style Product Hunt launches with generic names) to the same placeholder domain, `google.com`. `companies.domain` is `UNIQUE` in the schema, so writing it for the 2nd and 3rd company raised `psycopg2.errors.UniqueViolation` - uncaught, this killed `import_enriched_companies()`'s entire loop partway through, and since `_process_incoming_files()` only moves a file to `processed/` after the import function returns successfully, the poisoned file stayed in `data/clay/incoming_domain/` forever - the 60s background poller re-ran the exact same failing import on every tick.
**Fix:** Catch `psycopg2.errors.UniqueViolation` per-row inside `import_enriched_companies()`, call `conn.rollback()` (required - a raised statement aborts that row's own transaction, and each row already commits independently via `db.update_company_by_id()`), skip just that row, and continue - the same "nothing to write, not a stop-the-world error" treatment already given to a row with no domain value at all. Regression test added (`test_import_enriched_companies_skips_a_row_that_collides_with_an_existing_domain`).
**Live-verified:** re-ran the import against the real stuck file - 90 of 92 real domains written correctly, the 2 genuinely-mismatched rows correctly skipped (left `NULL`, not corrupted), file successfully moved to `processed/`, poller crash loop stopped.
**Talking point:** "The scoring model's honesty about not fabricating leads gets a lot of attention, but this is the same principle applied to data enrichment - Clay's real output can be wrong (a mismatched placeholder domain), and the fix isn't to silently trust it or crash the whole pipeline over it, it's to skip exactly the bad row and keep everything else moving. A crash-loop that quietly re-fails the identical error every 60 seconds forever is arguably worse than a clean crash - it needed a real user screenshot to notice the pattern at all."

## [Redesign v2, Tier 6] `utils/discord.py` shadowed the real `discord.py` pip library on import
**Symptom:** First attempt to start `uvicorn` after wiring in the new Discord bot module failed FastAPI startup entirely: `AttributeError: module 'discord' has no attribute 'Intents'`.
**Root cause:** This project's own Phase 9 webhook helper module was named `utils/discord.py` - the exact same importable name as the newly-added pip package `discord.py` (its package is literally `discord`, despite the PyPI/library name being `discord.py`). Every entry point (`api/main.py`, `python/full_pipeline_run.py`, `python/pipeline.py`) inserts `utils/` at the *front* of `sys.path`, so `import discord` anywhere in the process - including inside the new `utils/discord_bot.py`, which genuinely needed the real SDK - resolved to our own single-file module instead of the real library. This had been a latent landmine since Phase 9; it only surfaced once a real same-named pip package was added for the first time.
**Fix:** Renamed the file `utils/discord.py` → `utils/discord_webhooks.py` (and its test file to match, `tests/test_discord.py` → `tests/test_discord_webhooks.py`), and updated the 4 real call sites to `import discord_webhooks as discord` - every existing `discord.send_*()`/`discord.build_*()` call anywhere in the codebase is unchanged, since the local name `discord` still points at the same module inside each importing file.
**Live-verified:** `uvicorn` restarted cleanly, `/health` returned `200`, and the real bot (`discord.Client`) subsequently connected and handled a genuine user-uploaded attachment correctly.
**Talking point:** "This is a real, generalizable lesson for adding any new pip dependency to a project that already has its own module names: check for a name collision with anything already `sys.path`-inserted, especially in a project like this one where every entry point manually prepends `utils/`/`python/` rather than using a proper installable package layout."

## [Not a bug] Branch A/B/D returning identical company counts/names across repeated runs
**Symptom:** The user noticed 5+ manually-triggered full pipeline runs within a single evening all reported the exact same signal counts (22/35/20 for Branch A/B/D) and asked why "new" scraping kept returning identical output.
**Investigation, not assumed:** compared the real landed JSON files across 3 separate runs spanning hours - not just counts, the exact same company names in the exact same order every time. Then made a fresh, live, uncached call to the real SEC EDGAR API at that moment (`funding_edgar.search_form_d_filings()`) - it returned the exact same 22 companies again. Grepped all three branch modules for any caching layer (`lru_cache`, `requests_cache`, mocks, fixtures) - none exists anywhere.
**Root cause:** correct, expected behavior, not a bug. Branch A/B/D each query a fixed lookback window computed from `date.today()` (90/60/30 days respectively), called with no arguments from `full_pipeline_run.py` so always using those defaults. Re-querying the same real external database within a few hours of the last run will return the same real results unless a genuinely new Form D filing/job posting/PH launch matching the search landed in the intervening hours - unlikely for a narrow, niche search. This is the natural consequence of manually re-triggering "full cycle" test runs 5+ times in one evening rather than the intended daily-cron cadence.
**Talking point:** "A live re-call at the exact moment of investigation, not just re-reading old logs, was what actually proved this - the same real external system, queried again right then, gave the identical real answer. That's the difference between confirming a hypothesis and just restating it."

## [Redesign v2] Diagnosed why 0 qualified leads keeps happening - real root cause traced to Branch B's Adzuna noise, not scoring
**Symptom:** The user pushed back hard after 5+ full pipeline runs all produced 0 qualified leads out of 134 real companies, suspecting a scraping or scoring bug rather than accepting "the model is correctly strict" at face value.
**Investigation:** re-scored all 134 real companies live - max score 45 (Anduril Industries, `BOTH`), a large cluster capped at exactly 25 (TIMING-only). Traced *why* INTENT (the bucket that would let a company cross 70) was so rare: `SELECT count(*) FROM companies WHERE current_tool_mentioned IS NOT NULL` returned **0** across all 134 companies. Confirmed `merge_signals.merge_hiring_signals()` correctly writes this field when present - not a merge bug. Traced further upstream: `hiring_signals.py`'s Layer 2 (`hiring_ats_lookup.py`) only resolves a real Greenhouse/Lever board (with actual job-description text to scan for competitor mentions) for **2 of 35 real companies (5.7%)** this run - the other 33 fall back to Adzuna-only data, which has no description text at all, by a real external API limitation.
**Deeper root cause, found by manually classifying the real 33 unmatched company names:** 11 of 33 were staffing/recruiting agencies (Robert Half, Kelly Services, Jobot, RemX, Mondo, Tandym Tech, D24 Search, Beacon Talent, CultureMill Recruiting, HireTalent, Optomi) posting PM roles on behalf of undisclosed third-party clients - these can never represent real buying intent regardless of ATS-slug-guessing quality. Another 6 were large non-SaaS enterprises (J.P. Morgan, Mastercard, Citizens, Capital One, American Express, Oracle). `hiring_adzuna.py`'s Adzuna search uses the bare generic keywords `"Product Manager"`/`"Product Operations"` with **zero industry/company-type filtering** - unlike Branch A's EDGAR search, which explicitly requires `["software", "SaaS", "platform"]` in the filing text plus a fund-entity exclusion filter. Branch B had no analogous filter at all.
**Fix:** Added `_is_staffing_agency()` to `hiring_adzuna.py` (regex on common industry words + a curated list of known agency brand names, same documented-heuristic spirit as `funding_edgar.py`'s `FUND_NAME_PATTERN`), wired into `normalize_adzuna_results()` to drop staffing-agency jobs before they ever reach the ATS-lookup step. Deliberately did NOT attempt a "megacorp"/company-size filter - no real firmographic signal exists this early in the pipeline (only Clay's later per-company domain enrichment has that), and a brittle name-based guess would be exactly the kind of unfounded heuristic this project has consistently avoided elsewhere.
**Live-verified:** a real live Adzuna call before the fix returned 35 companies including all 11 known staffing agencies; after the fix, the identical live call returned 24 companies with zero staffing agencies present, while real companies (Anduril Industries, LevelTen Energy, etc.) remained.
**What this does NOT fix:** the megacorp/non-software noise (Oracle, Mastercard, etc.) and the underlying ~6% real ATS match rate on the companies that ARE legitimate SaaS candidates - both real, remaining gaps, documented rather than silently left unmentioned. 0 qualified leads may well continue until either real INTENT signal coverage improves further (better ATS matching, a refreshed `G2_API_TOKEN` to unblock Branch C) or the pipeline runs on a real daily cadence long enough to accumulate a genuine TIMING+INTENT overlap on the same company.
**Talking point:** "The user's instinct that '0 qualified leads out of 134 companies doesn't make sense' was the right instinct to push on, even though the scoring model itself turned out to be correct - the real bug wasn't in scoring, it was three layers upstream, in what Adzuna's bare keyword search was letting through unfiltered. Tracing a symptom all the way to its real root cause instead of stopping at the first plausible-sounding explanation is exactly what 'don't guess or assume' means in practice."
